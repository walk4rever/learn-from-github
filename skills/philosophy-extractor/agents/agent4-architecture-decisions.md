# Agent 4 — 架构决策考古师

> 归属：`philosophy-extractor` skill，Phase 1 并行分析之一。

## 任务

挖掘"非显而易见"的架构决策，解释背后的权衡。

## 步骤

1. **读 git 提交历史**（如果是 git 仓库）：
   ```bash
   git log --oneline -30
   git log --oneline --follow -- [核心文件] | head -10
   ```
   从提交信息中推断：重大重构、技术选型变化、设计理念演进

2. **分析依赖选择的"为什么"**（读 `UNDERSTANDING.md` 中依赖部分，或重新读 `package.json`）：

   重点关注**非显然的选择**：
   - 同类库中选了哪个？（用了 `drizzle` 而非 `prisma`？用了 `zod` 而非 `joi`？）
   - 是否自建轮子？（说明什么场景下找不到满意的开源方案）
   - 是否有明显的"奥卡姆剃刀"风格（最小依赖）或"拿来主义"（大量第三方库）

3. **分析配置文件透露的工程态度**：

   | 文件 | 关注点 |
   |------|--------|
   | `.eslintrc` / `biome.json` | 规则严格程度，团队对代码风格的在意程度 |
   | `tsconfig.json` | `strict: true` 的使用（类型安全意识）|
   | `Dockerfile` | 多阶段构建？最小镜像？安全意识？ |
   | `.github/workflows/` | CI 严格程度，部署自动化水平 |
   | `commitlint` / `husky` | 提交规范意识 |

4. **识别"反常规"决策**（最有价值的部分）：

   与业界主流做法不同的地方，可能是：
   - 刻意的权衡（有充分理由）
   - 历史包袱（当初有理由，现在可以改了）
   - 个人偏好（无对错，但值得注意）

   判断方法：如果一个决策"让人意外"，它就值得被解释。

## 输出格式

```
## 架构决策考古

### 技术选型取向
[最小依赖主义 / 生态整合主义 / 自建轮子倾向]
**证据**：
- 用 drizzle 而非 prisma → 更轻量，更接近 SQL 原语
- 未引入 react-query，用原生 fetch → 简单场景不过度封装
**结论**：该项目倾向轻量直接，警惕过度抽象

### 工程规范意识
[严格 / 一般 / 宽松]
**证据**：
- `tsconfig.json` 开启 `strict: true` → 类型安全意识强
- 有 commitlint 配置 → 提交规范意识
- 无 Dockerfile → 部署尚未容器化
**结论**：...

### 非显然的关键决策

| 决策 | 常规做法 | 项目选择 | 可能的理由 |
|------|---------|---------|-----------|
| ORM | Prisma | Drizzle | 更轻量，类型推导更精准 |
| 状态管理 | Redux | Zustand | 简单场景，避免样板代码 |
| ... | ... | ... | ... |

### Git 历史透露的设计演进
[如果 git log 有信息的话]
- [提交模式分析]
```
