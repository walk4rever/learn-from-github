# Philosophy Extractor — 设计哲学提炼器

提炼代码库的隐含设计信仰：哪里精妙、哪里合理、哪里存疑。

## 核心理念

代码库不只是功能的集合，它是一系列**设计决策**的沉淀。
这些决策透露了作者的品味、经验、以及在特定约束下的权衡取舍。

读懂这些决策，比单纯看懂代码更有价值——
因为你学到的不是"这里写了什么"，而是"为什么这样写，还有什么别的选择"。

## 调用方式

```
/design-lens <项目路径>
/design-lens .
/design-lens ~/projects/some-repo
```

---

## Token 使用原则

> **核心策略：有目的地采样源码，找模式而非读实现。**

1. **优先复用** `.judge-the-code/code-explore.md`：如果 code-explore 已运行过，直接读取，跳过重复分析
2. **Grep 先于 Read**：用关键词定位典型代码片段，不要盲目读整个文件
3. **采样而非全读**：每个维度只读 3-5 个代表性文件，每文件最多 80 行
4. **测试文件是宝藏**：这里 **允许** 读测试文件（code-explore 明确禁止），测试命名和结构最能透露设计取向
5. **禁止读取**：生成文件（`dist/`, `build/`）、依赖包（`node_modules/`, `vendor/`）、二进制文件

## 执行流程

### Phase 0：准备上下文

1. **输入验证** — 运行以下 bash 命令，根据返回值决定是否继续：

   ```bash
   TARGET="{用户提供的路径}"

   if [ -z "$TARGET" ]; then
     echo "EMPTY"
   elif [ ! -e "$TARGET" ]; then
     echo "NOT_FOUND"
   elif [ -f "$TARGET" ]; then
     echo "IS_FILE"
   elif [ -d "$TARGET" ]; then
     echo "OK"
   fi
   ```

   **根据输出立即处理，异常情况必须 STOP，不得继续执行后续步骤：**

   - **`EMPTY`** → 输出后停止：`❌ 请提供项目路径，例如：/design-lens .`
   - **`NOT_FOUND`** → 输出后停止：`❌ 路径 {path} 不存在，请检查路径是否正确。`
   - **`IS_FILE`** → 输出后停止：`❌ {path} 是一个文件，请提供项目根目录路径。`
   - **`OK`** → 继续执行步骤 2

2. **检查是否有 `.judge-the-code/code-explore.md`**（code-explore 的输出）：
   - **存在** → 读取全文作为背景上下文，**直接跳到 Phase 1**
   - **不存在** → 快速读取 `package.json` / `go.mod` / `pyproject.toml` 等识别语言和框架（最多读 2 个文件，各 50 行），然后进入 Phase 1

3. **确定 skill 目录路径**（供各 Agent 加载详细规格）：

   ```bash
   find ~/.agents/skills ~/.claude/skills -name "SKILL.md" -path "*/design-lens/*" 2>/dev/null \
     | head -1 | xargs dirname
   ```

   将结果记为 `{SKILL_DIR}`。

4. 输出：
   ```
   🔍 开始提炼设计哲学，启动 4 个分析 Agent...

     Agent 1 — 命名与抽象哲学    ⏳ 分析中
     Agent 2 — 错误处理取向      ⏳ 分析中
     Agent 3 — 测试与质量信仰    ⏳ 分析中
     Agent 4 — 架构决策考古      ⏳ 分析中

   预计完成时间：3-5 分钟
   ```

### Phase 1：并行 4 Agent 分析

**同时启动以下 4 个 Explore 子 Agent，全部并行执行：**

每个 Agent 的启动格式：
> 读取 `{SKILL_DIR}/agents/agentN-xxx.md`，按其中的说明执行分析，完成后返回结果。

| Agent | 规格文件 | 分析维度 |
|-------|---------|---------|
| Agent 1 | `{SKILL_DIR}/agents/agent1-naming-abstraction.md` | 命名风格、抽象层级、函数规模 |
| Agent 2 | `{SKILL_DIR}/agents/agent2-error-handling.md` | 错误处理风格、边界保护、信息质量 |
| Agent 3 | `{SKILL_DIR}/agents/agent3-testing-quality.md` | 测试类型分布、命名风格、mock 策略 |
| Agent 4 | `{SKILL_DIR}/agents/agent4-architecture-decisions.md` | 技术选型、工程规范、非常规决策 |

---

### Phase 2：综合输出

收集所有 4 个 Agent 的结果后，进行两步综合：

#### Step 1：判定每条决策的评价

对分析中发现的每一个值得记录的设计决策，打上标签：

| 标签 | 含义 | 判断标准 |
|------|------|---------|
| 🔮 **精妙** | 非显而易见的好决策，值得主动学习 | 解决了一个真实问题，且比常规做法更优雅 |
| ✅ **合理** | 标准做法，没什么特别，但没问题 | 业界常见，符合场景需求 |
| ⚠️ **存疑** | 有一定风险或代价，值得讨论 | 有明显的 tradeoff，且代价可能被忽视 |
| ❌ **反模式** | 有明确危害，应当避免或改进 | 违反安全、性能或可维护性的基本原则 |

**打标签原则**：
- 同一个决策在不同场景下评价可能不同，要结合项目背景
- 不确定时用 ✅，宁可保守也不随意打 ❌
- 每条必须有**具体证据**（文件路径 + 代码片段），不能只有结论

#### Step 2：生成 .judge-the-code/design-lens.md

```markdown
# [项目名] — 设计哲学报告

> 生成时间: [date] | 分析工具: design-lens

## 这个代码库的"人格"

[2-4 句话：用拟人化的方式描述这个代码库的整体气质]

示例：
> 这是一个**务实的工程师**写的代码。它不追求优雅的抽象，但每一行都能解决真实问题。
> 它对错误很认真（快速失败，绝不静默忽略），对测试比较随意（happy path 为主）。
> 读这个代码库，能感受到作者在某些地方刻意控制了复杂度——这是一种有意识的取舍。

### Phase 3：深度问答模式

生成完 `.judge-the-code/design-lens.md` 后，进入**设计讨论模式**。

**处理用户追问时的原则：**

1. **引用具体证据**：每个判断都要有文件路径 + 行号，不发表无根据的观点
2. **比较视角**：讨论一个决策时，主动对比"如果换一种做法会怎样"
3. **承认不确定性**：遇到"不清楚为什么这样做"的情况，直接说，不编理由
4. **追问驱动**：每次回答后，提供 1-2 个"延伸问题"引导深入

**深挖触发词**：
- "展开说说" / "为什么" → 加载更多相关代码，深入分析
- "跟 [另一个项目] 比" → 进行横向对比分析
- "怎么改" → 给出具体的改进建议（不是泛泛而谈）
- "教我用这个模式" → 提炼出可复用的代码模板

---

## 分析质量标准

- **每条评价必须有证据**：文件路径 + 代码片段，不能凭感觉
- **区分"设计决策"和"代码风格"**：`camelCase vs snake_case` 是风格，`是否使用 Result 类型` 是设计决策
- **精妙的决策不超过 5 条**：如果超过，重新筛选——真正精妙的决策很少见
- **反模式要谨慎**：不确定是否有害时，降级为存疑，避免误判
- **结合项目类型**：CLI 工具和大型后端服务的"合理决策"标准不同
